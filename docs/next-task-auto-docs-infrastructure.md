## Problem: No Infrastructure for Auto-Built Docs

The `working-with-claude-code` skill is the canonical example of a skill whose reference content is **generated**, not hand-authored. The 126 files in `skills/working-with-claude-code/references/` are fetched from `code.claude.com` by a script. Without infrastructure to manage this:

1. **Docs go stale** — no mechanism to update them on any schedule
2. **One-off process** — the update is tribal knowledge (find the script, run it manually)
3. **No convention** — as we add more skills that need built docs, everyone does it differently
4. **`_reference/` is a smell** — we used an external cloned repo as a scaffolding tool; the right approach is infrastructure inside this repo

This pattern will recur. Any skill that wraps external documentation (Claude Code docs, an SDK's API reference, a framework's docs) will need the same kind of "fetch + clean + commit" workflow.

## What Needs to Be Built

### 1. A root-level build/update system

Pick one approach (Makefile is probably the right call given this is a mixed-language repo with no existing package.json):

```makefile
# Makefile

update-working-with-claude-code:
	node skills/working-with-claude-code/scripts/update_docs.js

update-all: update-working-with-claude-code
	@echo "All auto-docs updated"
```

Or a `scripts/update.sh` shell script that discovers and runs all `skills/*/scripts/update*.{js,sh}` files.

### 2. A convention for skills with auto-built docs

Document (in CONTRIBUTING.md or CLAUDE.md) the convention:

- Skills with auto-built reference docs put their build script at `skills/<name>/scripts/update.js` (or `update.sh`)
- The script is self-contained: it writes to `skills/<name>/references/`
- The script exits non-zero on failure
- After running, generated files are committed to the repo (docs are content, not build artifacts)
- The root-level runner discovers scripts by globbing `skills/*/scripts/update*.{js,sh}`

### 3. Optionally: a GitHub Actions workflow

For each skill with a build script, add a scheduled workflow to keep docs fresh:

```yaml
# .github/workflows/update-docs.yml
on:
  schedule:
    - cron: '0 4 * * 1'  # Weekly Monday 4am UTC
  workflow_dispatch:       # Allow manual trigger
```

This is optional but high-value for the `working-with-claude-code` skill specifically, since Claude Code releases ship frequently.
