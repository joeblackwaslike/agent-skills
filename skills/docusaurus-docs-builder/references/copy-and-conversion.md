# Copy and Conversion

Docs copy should be useful first, persuasive second. The persuasion comes from fast clarity.

## Homepage Formula

```text
Headline: outcome for a specific audience
Subhead: pain -> mechanism -> proof
Primary CTA: get first success
Secondary CTA: see examples or read concepts
Hero visual: demo, architecture, screenshot, or terminal proof
```

## Page Intro Formula

Each docs page starts with:

1. What this page helps you do.
2. When to use it.
3. What you need before starting.

Keep it under 80 words unless the topic is inherently risky.

## CTA Rules

- "Start the quickstart" beats "Get started".
- "Install the CLI" beats "Learn more".
- "Configure GitHub auth" beats "Next".
- Every CTA should tell the user what happens next.

## Anti-wall-of-text Rules

- One idea per paragraph.
- Use bullets for choices and requirements.
- Use tables for options.
- Use tabs for variants.
- Use diagrams for flows.
- Move deep explanation below the working example.

## Voice

Use direct, technical, confident language:

- "Run this command."
- "This creates three files."
- "Use this when..."
- "Avoid this if..."

Avoid:

- "Simply"
- "Just"
- "Seamlessly"
- "Robust and scalable"
- "Unlock the power of"
- "In today's fast-paced world"

## Trust Builders

Use proof close to claims:

- Screenshots near UI claims.
- Benchmarks near performance claims.
- Code examples near API claims.
- Architecture diagrams near system claims.
- Troubleshooting near setup claims.
