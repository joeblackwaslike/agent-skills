# Project Docs Template (proven scaffold)

A complete, battle-tested Docusaurus docs scaffold for a TypeScript project — the exact
structure shipped by [`spinup-ts`](https://github.com/joeblackwaslike/spinup-ts) when it
generates a new repo. Use it two ways:

- **Bootstrapping a new project** — drop the whole tree in, then fill the blanks.
- **Revitalizing an existing repo** — adopt the taxonomy and backfill pages from real code,
  README, and history instead of leaving a thin `docs/intro.md`.

The scaffold is opinionated on purpose. It gives every project the same shape (Getting
Started → User Guide → Developer Guide → Architecture → Reference → API → Research →
Post-mortems → Changelog), so readers who know one repo already know the next. Every page
ships as a skeleton with placeholder tokens and `Replace with…` prompts — the agent's job
is to **change the details to match the current project**, not to invent a new structure.

> This is the concrete companion to `references/information-architecture.md`. IA explains
> *why* docs are organized around user jobs; this file is the *ready-to-paste* realization
> of that principle. When a user says "scaffold docs" or "set up docs like spinup-ts",
> start here.

## Placeholder tokens

Every page uses literal tokens so a generator (or the agent) can find-and-replace them.
When adapting, replace **all** occurrences across every file:

| Token | Meaning | Example |
| --- | --- | --- |
| `__PROJECT_NAME__` | npm package / repo name | `my-lib` |
| `__PROJECT_SLUG__` | CLI binary or env-var prefix (often == name) | `my-lib` |
| `__DESCRIPTION__` | one-line value proposition | `Type-safe retry helpers` |
| `__GITHUB_HANDLE__` | GitHub org/user | `joeblackwaslike` |
| `__AUTHOR__` | display name for footer/credits | `Joe Black` |
| `__LICENSE__` | SPDX id | `MIT` |
| `__NODE_VERSION__` | minimum Node major | `20` |
| `__YEAR__` | copyright year | `2026` |

After replacement, **delete the `Replace with…` prompts and any page that doesn't apply**
(see [Section pruning](#section-pruning) below). A scaffold left full of "Replace with…"
is worse than no docs — it signals abandonment.

`autogenerated` entry.

### docusaurus.config.ts

`routeBasePath: '/'` makes docs the whole site (no separate landing page to maintain).
TypeDoc auto-generates the API section; `docusaurus-plugin-llms` emits `llms.txt`. The
`url`/`baseUrl`/`organizationName`/`projectName` block is the GitHub Pages deploy contract.

```ts
import type { Config } from '@docusaurus/types';
import type * as Preset from '@docusaurus/preset-classic';

const config: Config = {
  title: '__PROJECT_NAME__',
  tagline: '__DESCRIPTION__',
  favicon: 'img/logo.svg',

  url: 'https://__GITHUB_HANDLE__.github.io',
  baseUrl: '/__PROJECT_NAME__/',
  organizationName: '__GITHUB_HANDLE__',
  projectName: '__PROJECT_NAME__',
  trailingSlash: false,

  onBrokenLinks: 'throw',
  onBrokenMarkdownLinks: 'warn',

  i18n: { defaultLocale: 'en', locales: ['en'] },

  plugins: [
    [
      'docusaurus-plugin-typedoc',
      {
        entryPoints: ['./src/index.ts'],
        tsconfig: './tsconfig.json',
        out: 'api',
        sidebar: { categoryLabel: 'API Reference' },
      },
    ],
    [
      'docusaurus-plugin-llms',
      { generateLLMsTxt: true, generateMarkdownFiles: true },
    ],
  ],

  presets: [
    [
      'classic',
      {
        docs: {
          sidebarPath: './sidebars.ts',
          editUrl: 'https://github.com/__GITHUB_HANDLE__/__PROJECT_NAME__/edit/main/',
          routeBasePath: '/',
        },
        blog: false,
      } satisfies Preset.Options,
    ],
  ],

  themeConfig: {
    navbar: {
      title: '__PROJECT_NAME__',
      items: [
        { type: 'docSidebar', sidebarId: 'mainSidebar', position: 'left', label: 'Docs' },
        {
          href: 'https://github.com/__GITHUB_HANDLE__/__PROJECT_NAME__',
          label: 'GitHub',
          position: 'right',
        },
      ],
    },
    footer: {
      style: 'dark',
      copyright: `Copyright © __YEAR__ __AUTHOR__. Built with <a href="https://docusaurus.io">Docusaurus</a>.`,
    },
    colorMode: {
      defaultMode: 'light',
      disableSwitch: false,
      respectPrefersColorScheme: true,
    },
    // Algolia DocSearch — apply at https://docsearch.algolia.com/apply/
```

## Adaptation playbook

The scaffold is the starting line, not the finish. After dropping it in, **change the
details to match the real project.** Do this work — do not leave placeholders behind.

### Bootstrapping a new project

1. Replace every [token](#placeholder-tokens) across all files.
2. Write the `intro.md` value proposition for real — problem, solution, one honest
   differentiator. This is the page everyone reads first.
3. Fill `getting-started.md` with the **simplest call that actually runs**. Copy-paste it
   into a scratch file and execute it before committing.
4. Leave Architecture and Research pages as honest stubs if the project is brand-new, but
   keep the headings — they create slots that get filled as the project matures.
5. Prune sections that don't apply (see below).

### Revitalizing an existing repo

1. Inventory what already exists: README, inline docs, `examples/`, tests, CLI `--help`,
   exported types. These are your source material — mine them, don't ignore them.
2. Map existing content onto the taxonomy using the README split rules in
   `references/information-architecture.md` (hero → intro, install → installation,
   features → user-guide/features, diagrams → architecture, options → reference).
3. Generate the API section from real exports — point `entryPoints` at the true
   `src/index.ts` and let TypeDoc do it.
4. Fill `architecture/*` from the code that already exists, not aspirations. The
   Constitution should describe principles the code **already follows**; if it doesn't,
   that's a finding, not a doc.
5. Backfill `post-mortems/` with any real incidents worth remembering — this directory is
   what makes the structure compound in value over time.
6. Delete every remaining "Replace with…" line before you ship.

### Section pruning

Not every project needs every page. Cut decisively:

| Page / section | Keep when | Drop when |
| --- | --- | --- |
| `reference/cli.md` | Project ships a CLI binary | Library-only |
| `reference/hooks.md` | There's a plugin/hook system | No extensibility surface |
| `architecture/ui.md` | There's a rendered UI | Headless lib, CLI, server |
| `research/*` | You did real market/demand research | Internal tool, obvious niche |
| `developer-guide/*` | You expect outside contributors | Solo/private project |
| `post-mortems/` | Production system with incidents | Pre-release experiment |

When you drop a page, remove its `sidebars.ts` entry too — a sidebar link to a deleted
file fails the `onBrokenLinks: 'throw'` build.

### Verification

Before declaring docs done, run the skill's standard completion gate (`references/quality-gate.md`)
plus these template-specific checks:

- `pnpm docs:build` succeeds (TypeDoc generates `docs/api/`, no broken links).
- Zero `Replace with…` / `__TOKEN__` strings remain: `grep -rE 'Replace with|__[A-Z_]+__' docs/`.
- The `getting-started.md` example runs as written.
- Every `sidebars.ts` entry resolves to a real file.
